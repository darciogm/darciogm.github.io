    Data Acquisition — Audesp¶

    Source description¶

    Audesp (Auditoria Eletrônica de Órgãos Públicos) is the electronic auditing system maintained by the Tribunal de Contas do Estado de São Paulo (TCE-SP). It receives structured data from all 645 municipalities in São Paulo state on procurement processes (licitações), contracts, spending commitments, and budget execution.

    The procurement data used here covers bid-level records for all competitive and non-competitive processes published through the Audesp portal, spanning 2018 through December 2024.

    Acquisition and conversion¶

    Raw data were obtained as semicolon-delimited CSV files organized by month, then batch-converted to Parquet using csv_to_parquet.py.

    Why Parquet?

    • ~80% compression vs. CSV (storage and I/O)
    • Columnar format enables reading only needed columns (4 of 21 in our case)
    • Typed columns eliminate repeated type inference on every load

    The converter uses:

    • Chunked reading calibrated to available RAM (psutil)
    • Parallel processing (ProcessPoolExecutor)
    • Encoding detection (chardet) with latin-1 fallback
    • Resumable design (skips already-converted files)

    Data quality issues¶

    1. Column name encoding¶

    Some monthly Parquet files wrap column names in literal quotes:

    Schema shows: "Município": large_string
Expected:     Município: string

    Resolution: _resolve_columns() in build_nll_panel.py strips quotes and matches case-insensitively against the expected column list.

    2. Modality name heterogeneity¶

    408 distinct modality strings found across all files. Examples of the same modality:

    "Pregao Eletronico"
"Pregão Eletrônico"
"PREGÃO ELETRÔNICO"
"Pregão Eletrôcnico - Compras Gov."
"pregão eletronico"

    Resolution: _classify_modality() uses accent-stripped, lowercased substring matching to map all variants to 12 canonical categories. Coverage: 99.97%.

    3. Memory constraints¶

    The 2018 annual file (19.7M rows) expands to 9.5 GB in memory when all columns are loaded as strings. On a 15 GB WSL instance, this causes OOM kills.

    Resolution: Batch processing reads only 4 needed columns via pyarrow.ParquetFile.iter_batches(batch_size=1_000_000), keeping peak memory at ~1.5 GB.
